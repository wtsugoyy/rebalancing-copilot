# Rebalancing Copilot: Agent Guide

Offline portfolio attribution, rebalancing, and history, with an LLM chat layer that routes to
deterministic tools and never computes anything itself.

## How to run

```powershell
# Full offline stack (native Ollama+GPU, containerised app/langfuse/supermemory)
powershell -File scripts\start_stack.ps1
# dashboard http://localhost:8501 · langfuse http://localhost:3000

# local
python -m pip install -r requirements.txt ; streamlit run app.py

# tests (run before any commit)
$env:LANGFUSE_ENABLED=0                   # unless the stack is up: see below
python -m pytest -q                       # full suite (~35s), incl. regression gates
python -m pytest tests/test_soak.py -q    # determinism/no-silent-failure soak
```

**Always set `LANGFUSE_ENABLED=0` when the stack is down.** Tracing fails open, so the tests
pass regardless, but the SDK retries against an absent server and a 35s run becomes ~10min.
If a run appears to hang around 92%, that is this, not a broken test.

Ollama runs **natively** (GPU); `scripts\start_stack.ps1` starts it bound to `0.0.0.0` so
containers reach it via `host.docker.internal`. Models: `ollama pull qwen3:14b` (agent) and
`ollama pull qwen2.5:3b` (Supermemory's extraction agent).

## All data in this repo is synthetic

The 12-fund universe (`ZZ`-prefixed ISINs, invented names), the weights, the return series, the
yields, and the demo holdings are generated by `scripts/make_synthetic_fixtures.py` from a
seeded RNG. `config.py` is the source of truth for the universe; the fixtures are derived from
it, never transcribed alongside it.

**The golden fixtures are regression baselines, not an oracle.** Their expected values were
produced by running this engine over the synthetic inputs. They catch drift from a refactor;
they cannot prove a metric definition is correct, because the engine computed them. Do not
describe them as validation. If a change to `engine/` moves these numbers, that is the gate
working. Regenerate only once you've decided the new behaviour is right.

## Agent stack (qwen3:14b + LangGraph + Langfuse + Supermemory)

```
recall ─▶ model ─┬─(tool_calls)─▶ deterministic tools ─┐
                 │                                      └─▶ back to model (capped)
                 └─(no tool_calls)──────────────────▶ remember ─▶ END
```

- **Model: `qwen3:14b`.** Reasoning ("thinking") **and** native tool calling. Runs on the
  **native Windows Ollama** (100% GPU via ROCm `gfx1200` on the RX 9060 XT). Docker on Windows
  cannot pass through an AMD GPU, so Ollama must NOT be containerised.
- **DeepSeek-R1 was evaluated and rejected.** Official Ollama R1 builds ship no tool-calling
  template: asked for BAL's volatility it ignored the tool, hallucinated the ticker "BAC", and
  invented a method. It would fabricate financial figures here.
- **Harness: LangGraph** (`agent/graph.py`). `agent/agent.py` keeps the older simple loop and
  is still used for `check_ollama()`/`TOOL_SCHEMAS`.
- **Tracing: self-hosted Langfuse v2** (`agent/tracing.py`), telemetry off.
- **Memory: self-hosted Supermemory** + local SQLite FTS fallback (`agent/memory.py`).

## Rebalancing engine (engine/optimizer.py)

Mean-variance, HARD volatility cap: maximise `w·mu` s.t. `sqrt(w'Σw) <= σ*`, `Σw=1`, per-fund
min/max bounds. Conventions:
- **Value 1** (per-fund expected return) = `(prod(1+r))^(252/n_obs) - 1` (trading-day CAGR)
- **Value 2** (per-fund volatility) = `std(daily, ddof=1) * sqrt(252)`
- Σ = `cov(daily, ddof=1) * 252`; blank return cells = 0.

SLSQP, deterministic multi-start (the min-vol point is always a feasible start).
`_round_weights` rounds to whole percents while preserving `Σw=1` and the bounds.

`tests/test_optimizer.py` gates two different things. Keep them distinct:
- *Regression*: the recorded blocks (σ 3.5/4.5/5.5%) still reproduce. Only meaningful against
  the current fixture.
- *Structural*: bounds honoured, hard cap never breached, deterministic across runs, higher
  target never lowers return. These are real properties and hold for any input.

The UI has four tabs: Overview, **Rebalancing Engine** (exclusive), History, Benchmark.
Benchmarks: presets + per-period % returns saved to the `benchmarks` table (latest = active).
Snapshot sources counting as "current": `user_committed`, `seed`, `engine`
('candidate_preview' never). Beta vs a flat-rate benchmark is undefined, so tools return null +
reason; YTM/sector need factsheet data, so tools say so; top-5 holdings look-through covers
3/12 funds (`sample/top5_holdings.json`).

## Tool surface (22 tools)

Core: `list_portfolios`, `get_current_portfolio`, `get_history`, `attribution`, `vol_vs_target`.

**Analyst toolkit** (`engine/analytics.py`, deterministic, reconciles to `perf.py`):
`periodic_returns` (per year/quarter/month), `volatility_trend` (rising/falling; reports
"right now" AND whole-period separately, do not conflate them), `attribution_period` (date
range), `fund_contribution`, `drawdown_periods`, `compare_portfolios`, `correlations`.

**Fenced sandbox** (`engine/sandbox.py`): `run_analysis` runs model-written pandas/numpy over
the loaded data for open-ended questions. AST **allowlist** (not blocklist): no imports, no
I/O, no dunder/attribute escapes, restricted builtins, wall-clock timeout. Results are stamped
`advisory: true` and the code is surfaced for audit. They are NOT regression-tested like
`perf.py`. `tests/test_sandbox.py` is the security boundary; if a "blocked" case ever starts
passing, the sandbox is compromised.

The model was never "dumb", it was tool-starved. If it can't answer something, the fix is
almost always a missing tool, not a bigger model.

## LLM backend is pluggable (agent/llm.py)

`LLM_PROVIDER=ollama` (default, offline, local GPU) or `LLM_PROVIDER=gateway` (any remote
OpenAI-compatible endpoint; bearer in `.env`/`GATEWAY_BEARER`). Gateway mode **sends fund data
off this machine**. That is deliberate; never flip it silently. Probed differences: the gateway
returns tool-call `arguments` as a **JSON string** (Ollama: dict), needs `tool_call_id` on tool
results (Ollama: `tool_name`), and its `reasoning` field is always empty. Only the transport
differs; engine/tools/tests are shared.

## PROMPT ORDERING IS LOAD-BEARING. NEVER append anything after SYSTEM_PROMPT

Build the system prompt as **context, then memories, then SYSTEM_PROMPT**. The rules must be
last. Measured on a gateway model (`local-main`, 22 tools): rules-first = **0/6** native tool
calls; context-first = **6/6**. Even innocuous filler appended after the rules degraded it
(1/3). Appending silently turns tool calls into prose JSON, i.e. ungrounded numbers on
financial data. qwen3:14b tolerated the bad order; local-main does not. Pinned by
`tests/test_context.py::test_rules_come_last_in_system_prompt`.

Known-open (gateway only): `local-main` over-reaches for `run_analysis` instead of the specific
tools. SYSTEM_PROMPT rule 9 + a hardened sandbox schema route it; if that proves insufficient,
hide `run_analysis` from the gateway provider's tool list rather than let advisory numbers
answer what a dedicated tool covers.

## Architecture (keep these invariants)

- **`app.py` is the ONLY file that may import `streamlit`.** Everything in `engine/`, `store/`,
  `agent/` is UI-framework-agnostic pure Python, so a FastAPI reskin wraps them unchanged.
- **The LLM never computes.** The model may only *select* a tool; every number comes from
  deterministic `engine/` code. Reasoning about what numbers *mean* is encouraged; producing
  numbers is forbidden. Guarded by `tests/test_graph.py`.
- **Observability and memory must never break the product.** Langfuse and Supermemory failures
  degrade to no-ops (`tests/test_graph.py::test_agent_survives_dead_memory_and_tracing`).

## Upstream bugs we work around (do not "fix" by loosening these)

- **`langfuse` SDK must stay `<3`.** We self-host `langfuse/langfuse:2`; v3/v4 SDKs expect
  `organization`/`metadata` fields on `/api/public/projects` that the v2 server doesn't return,
  so `auth_check()` raises a pydantic `ValidationError` and tracing silently turns off. Bump
  the server image before relaxing the pin in `requirements.txt`.
- **Supermemory `0.0.3` recall is broken.** Writes succeed, documents reach `status=done`, and
  the memory agent extracts memories, but `/v3/search` and `/v4/search` always return
  `total: 0` (verified with/without `containerTag`, `threshold: 0`). `agent/memory.py`
  therefore dual-writes to Supermemory and reads from a local FTS5 index. It will prefer
  Supermemory automatically once upstream returns hits.
- Supermemory's memory agent needs a **non-thinking** model for extraction (we point it at
  `qwen2.5:3b`); a thinking model's `<think>` blocks break its structured output.
- **No silent failures.** Bad input raises a typed error from `engine/validate.py`; the UI
  renders it as a banner, the agent returns it as a structured `{"error": ...}`.
- **History is append-only.** Never add an UPDATE/DELETE path; corrections are new rows.

## Metric conventions (deliberate, do NOT "fix")

`engine/perf.py` carries two annualisation conventions that look inconsistent and are kept on
purpose; mixing them silently would be worse than the asymmetry:

- **Annualised return** uses `365 / elapsed-calendar-days` (CAGR).
- **Annualised volatility** uses a hardcoded `√252` (trading days).
- `STDEV` is sample stdev (`ddof=1`).
- Blank fund-return cells count as `0` in the weighted sum.
- First observation: `rf_period = 0`, `excess = 0`.

The rf lookup falls back to `RF_FALLBACK` (3.3% p.a.) for dates beyond the yield series. The
fixture's `rf_lookup_cutoff` deliberately stops short of the last NAV date so that fallback
path stays under test.

## Regenerating fixtures

```powershell
python scripts/make_synthetic_fixtures.py
```

Regenerates `tests/fixtures/*.json` and `sample/*`. See the caveat above: a diff here means
engine behaviour changed.

## Fund universe

12 synthetic funds, 6 portfolios (SC, SC+, BAL, BAL+, ADV, ADV+). Defined in `config.py`
(`FUND_UNIVERSE`, `FUND_NAMES`, `DEFAULT_WEIGHTS`); `DEFAULT_WEIGHTS` seeds the "current"
allocation on first data load.
